# ADR-022: Regenerate Android template for edge-to-edge (system bars)

## Status

Accepted

## Date

2026-07-02

## Context

On an Android tablet (Xiaomi, latest Android version), dark strips appeared
above and below the WebView content — exactly where the OS status bar (top) and
navigation bar (bottom) live. The WebView was drawn in the safe area between
them, while the system bars rendered on top of the activity's `windowBackground`
(a dark color), producing visible dark bands.

Well-behaved native Android apps use **edge-to-edge**: system bars are
transparent, and UI is drawn underneath them. This is the standard since Android
15 (API 35) and is achieved by calling `enableEdgeToEdge()` in `onCreate()`.

### Root cause

The `tauri/gen/android/` template was generated by an older `tauri-cli` (< 2.6.3)
whose `MainActivity.kt` does **not** call `enableEdgeToEdge()`. Without that
call, the system bars are opaque and drawn over the activity background, not
over a transparent layer with content extending underneath.

PR [tauri-apps/tauri#13780](https://github.com/tauri-apps/tauri/pull/13780)
(merged in `tauri-cli` 2.6.3, July 2025) added `enableEdgeToEdge()` to the
canonical `MainActivity.onCreate` template. The project's `tauri` crate was
already 2.10.3 (`Cargo.lock`), but the **Android template** was stale because
`tauri android init` had not been re-run since the original generation.

Additional staleness in the old template:

- `compileSdk = 34` / `targetSdk = 34` — Google Play requires ≥ 35.
- Outdated dependencies: `webkit:1.6.1`, `appcompat:1.6.1`, `material:1.8.0`.
- Missing `androidx.activity:activity-ktx` (required by `enableEdgeToEdge()`).
- AGP 8.5.1 (canonical 2.9.6 ships 8.11.0).

### tauri-cli version discrepancy

The project has **two** tauri-cli installations:

- `cargo tauri` (via `cargo install tauri-cli`): **2.9.6** — includes
  `enableEdgeToEdge()` in the template.
- `tauri` / `npx tauri` (via `@tauri-apps/cli` npm, `tauri/package.json`):
  **2.5.0** — does **not** include it.

Regeneration **must** use `cargo tauri android init`. Using the npm-based CLI
(2.5.0) would regenerate the old template without edge-to-edge, defeating the
purpose. This discrepancy also explains why the old `BuildTask.kt` hardcoded
`node_modules/@tauri-apps/cli/tauri.js` (commit `0d2a1504`) — it was a
workaround for the stale npm CLI. After regeneration via `cargo tauri`, the
template substitutes `cargo` as the binary (verified against
`tauri-cli/src/mobile/init.rs`: the `bin_stem == "cargo-tauri"` branch in a
release build sets `binary = "cargo"`, `args = ["tauri", ...]`), making the
npm pinning obsolete.

## Decision

**Regenerate the entire `tauri/gen/android/` directory from the canonical
`tauri-cli` 2.9.6 template**, then restore the project-specific customizations
that the regeneration wipes.

Chosen over a manual 10-line patch (`enableEdgeToEdge()` + `activity-ktx`
dependency) because:

1. **Google Play compliance**: SDK 34 → 36 is a hard requirement, not a
   side effect. A manual edge-to-edge patch would leave the template
   non-compliant, requiring a separate PR.
2. **Dependency freshness**: `webkit`, `appcompat`, `material`, AGP — all
   updated to current versions in one step.
3. **Upstream alignment**: manual patching means permanent divergence from the
   canonical template; regeneration resets the baseline.

### Preserved customizations

The regeneration wiped five project-specific customizations (identified via
`git log` on `tauri/gen/android/` and a `diff -r` against the pre-regeneration
backup). Four were restored; one was intentionally dropped as obsolete.

| # | Customization | Commit | Restoration |
| --- | --- | --- | --- |
| 1 | **Zoom-fix** (`onStart`/`configureWebView`/`applyNativeScrollSettings`) — disables pinch-zoom and overscroll in the WebView for a native-app feel. | `f4ba855a` (#211) | Merged into the regenerated `MainActivity.kt` alongside `enableEdgeToEdge()`. |
| 2 | **Deep-link intent-filter** (`origa://` scheme) — VIEW/DEFAULT/BROWSABLE filter for the deep-link plugin. | `754c1c4c` | **Not manually restored — auto-injected.** `cargo tauri android init` does not include the filter in the generated manifest, but `cargo tauri android build` auto-injects it via the deep-link plugin (marked `<!-- DEEP LINK PLUGIN. AUTO-GENERATED. DO NOT REMOVE. -->`). The manual filter from commit `754c1c4c` is obsolete — it was originally added because the old template predated the plugin's auto-injection feature. |
| 3 | **Signing config** (`signingConfigs { create("release") { ... } }` reading `../keystore.properties`). | `b5a3679a` | Restored in `app/build.gradle.kts` + `import java.io.FileInputStream`. The canonical template has no signing block. Guarded by `if (keystorePropertiesFile.exists())` — inert when no keystore is present (local debug builds). |
| 4 | **Branded launcher icons** (mipmap PNGs in all densities). | `e1eb054c` | Restored from backup. `cargo tauri android init` generates **default Tauri** mipmaps, not branded ones — it does not regenerate from `tauri/icons/` (that is the `tauri icon` command's job). Verified by byte-size comparison: regenerated `ic_launcher.png` (mdpi) = 3377 bytes (default) vs branded 5025 bytes. |
| 5 | ~~`BuildTask.kt` node_modules pinning~~ — hardcoded `node` + `node_modules/@tauri-apps/cli/tauri.js`. | `0d2a1504` | **Not restored — obsolete.** The npm CLI path (`node_modules/...`) was absent from disk (`node_modules/` not installed), so the old `BuildTask` was non-functional for Android Studio builds without a prior `npm install`. The regenerated `BuildTask` uses `cargo` as the binary (substituted by `tauri-cli` based on `argv[0]`), which resolves to the fresh 2.9.6 CLI without any npm dependency. This fixes a latent bug. |

### BuildTask runtime verification note

The regenerated `BuildTask.kt` (cargo-based) was verified by **source
inspection** of `tauri-cli/src/mobile/init.rs` (the `cargo-tauri` binary-stem
branch) and by successful `gradlew :app:compileUniversalDebugKotlin` (which
compiles `buildSrc` including `BuildTask.kt`). Full runtime verification of
`BuildTask.assemble()` (which invokes `cargo tauri android android-studio-script`
during the Rust-`assemble` phase) requires a complete Android APK build or
device testing — not exercised in this PR.

## Alternatives Considered

### Manual patch: add `enableEdgeToEdge()` + `activity-ktx` to the existing template

- **Pros:** Minimal blast radius (~10 lines); no risk to existing customizations.
- **Cons:** SDK 34 → 36 (Google Play compliance) remains unaddressed,
  requiring a separate PR. Dependencies stay stale. Permanent divergence from
  the upstream template — every future `tauri android init` would need the same
  manual patch re-applied.
- **Rejected.** The user explicitly chose regeneration (with Tech Lead research)
  to address all three concerns (edge-to-edge + SDK compliance + dependency
  freshness) in one step.

## Consequences

### Positive

- **Dark system-bar strips eliminated.** `enableEdgeToEdge()` makes the status
  and navigation bars transparent, with UI drawn underneath them.
- **Google Play compliance:** `compileSdk`/`targetSdk` = 36 (was 34).
- **Dependencies refreshed:** `webkit:1.14.0`, `appcompat:1.7.1`,
  `material:1.12.0`, `activity-ktx:1.10.1`, AGP 8.11.0.
- **Latent BuildTask bug fixed:** the cargo-based `BuildTask` works without
  `npm install`; the old node_modules reference pointed at a non-existent path.
- **Windows fallback:** the canonical 2.9.6 `BuildTask.kt` includes
  `.exe`/`.cmd`/`.bat` fallback logic (PR #14146), improving Windows
  compatibility.

### Negative

- **Future `tauri android init` will wipe customizations again.** This is
  documented Tauri behavior — `gen/android/` is a generated directory. The four
  restored customizations (zoom-fix, deep-link, signing, icons) must be
  re-applied after any future regeneration. **Mitigation:** this ADR documents
  the full restoration procedure. Long-term mitigation: upstream the zoom-fix to
  the Tauri template (or convert it to a Tauri plugin) so it survives
  regeneration.
- **Signing config not exercised in this PR.** No local `keystore.properties`
  exists (CI provisions it). The config is structurally restored and
  compile-verified, but release-signing was not tested end-to-end.
- **Chromium WebView bug #441253216.** `env(safe-area-inset-*)` CSS returns `0px`
  in Android WebView before Chromium 140.0.7339.51. This means that if the app's
  CSS relies on `safe-area-inset-top`/`-bottom` to pad content away from the
  now-transparent system bars, the padding would be zero on older WebView
  versions. Not expected to affect the target device (Xiaomi, latest Android →
  fresh WebView ≥ 140), but documented for future reference. If it becomes an
  issue, the fix is a Tauri plugin that exposes native window insets to JS (see
  [issue #14240](https://github.com/tauri-apps/tauri/issues/14240)).

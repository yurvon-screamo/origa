# Процесс релизов Origa

## Обзор

Origa использует автоматизированную систему версионирования и релизов на базе GitHub Actions. Система поддерживает два типа релизов: стабильные (stable) для официальных релизов и разработческие (dev) для ежедневных сборок.

Версионирование происходит автоматически без необходимости ручного обновления версии в коде - просто создай тег для stable релиза, и система сама вычислит правильную версию для dev-сборок.

## Типы релизов

### Stable релизы (стабильные)

Stable релизы создаются при push тега в формате `v*.*.*` (например, `v0.2.0`).

**Характеристики:**
- Версия берётся напрямую из тега (v0.2.0 → версия 0.2.0)
- Создают перманентный релиз с тегом версии
- Не обновляются - каждая версия уникальна

**Артефакты:**
- **Windows**: `.exe` (установщик), `.msi` (MSI пакет)
- **Linux**: `.AppImage` (portable), `.deb` (Debian/Ubuntu)
- **macOS**: `.zip` (архив приложения)
- **Android**: `.apk` (Android приложение)
- **Docker**: Docker образ с тегом версии

### Dev релизы (разработческие)

Dev релизы создаются автоматически при каждом push в ветку `master`.

**Характеристики:**
- Версия = base.patch+1 + short sha (например `0.1.3-abc1234`)
- Обновляют существующий релиз с тегом `latest`
- Всегда содержат последнюю версию из master

**Артефакты:**
Те же что и для stable, но маркированы как "Latest Build"

## Версионирование

### Формат версии

**Stable:**
```
X.Y.Z
```
где:
- X - major версия (значительные изменения)
- Y - minor версия (новые функции)
- Z - patch версия (исправления ошибок)

**Dev:**
```
X.Y.Z-SHA
```
где:
- X.Y.Z - версия на основе последнего тега + 1 к patch
- SHA - первые 7 символов хеша коммита

### Алгоритм вычисления версии

1. **При push тега (stable релиз):**
   - Версия извлекается из имени тега
   - Пример: тег `v0.2.0` → версия `0.2.0`

2. **При push в master (dev релиз):**
   - Находит последний тег в формате `v*.*.*`
   - Инкрементирует patch версию на 1
   - Добавляет short SHA коммита
   - Пример: последний тег `v0.1.2` → dev версия `0.1.3-abc1234`

## Как создать релиз

### Создание stable релиза

> **Примечание:** CI автоматически запускается при push тега - никаких ручных действий в GitHub Actions не требуется.

```bash
# 1. Убедись что master актуален
git checkout master
git pull origin master

# 2. Создай тег с новой версией
git tag v0.2.0

# 3. Запуш тег в origin
git push origin v0.2.0

# 4. CI автоматически:
#    - Соберёт все артефакты для всех платформ
#    - Создаст релиз с тегом 0.2.0
#    - Загрузит артефакты в релиз
#    - Опубликует Docker образ
```

**Важно:** После создания stable релиза следующие dev-сборки будут базироваться на этой версии.

### Dev релизы

Dev релизы создаются автоматически и не требуют никаких действий:

```bash
# Просто работай с кодом и пуш в master
git add .
git commit -m "fix: что-то исправлено"
git push origin master

# CI автоматически создаст/обновит релиз "latest"
# с версией типа 0.2.1-abc1234
```

## CI/CD пайплайны

### Триггеры запуска
- Push в ветку master → Dev релиз (версия X.Y.{patch+1}-sha)
- Push тега v*.*.* → Stable релиз (версия из тега)
- Ручной запуск (workflow_dispatch)

### Workflow файлы

| Файл | Назначение |
|------|------------|
| `.github/workflows/version.yml` | Вычисление версии для всех типов релизов |
| `.github/workflows/tauri.yml` | Сборка десктоп приложений (Windows, Linux, macOS) |
| `.github/workflows/mobile.yml` | Сборка мобильных приложений (Android) |
| `.github/workflows/docker.yml` | Сборка и публикация Docker образа |
| `.github/workflows/reusable-release.yml` | Reusable workflow для создания релиза |

### Порядок выполнения

```
Push в master / Push тега
         ↓
    version.yml
    (вычисляет версию)
         ↓
    ┌────┴────┬─────────┬─────────┐
    ↓         ↓         ↓         ↓
 tauri.yml mobile.yml docker.yml  ...
    │         │         │         │
    └────┬────┴─────────┴─────────┘
         ↓
  reusable-release.yml
  (создаёт/обновляет релиз)
         ↓
     Релиз готов
```

**Детали:**
1. `version.yml` определяет тип релиза и вычисляет версию
2. Параллельно запускаются сборки для всех платформ
3. Каждый workflow передаёт версию в `reusable-release.yml`
4. Первый workflow создаёт релиз, остальные добавляют артефакты

## Артефакты релиза

### Windows
- **NSIS installer** (`.exe`) - классический установщик Windows
- **MSI installer** (`.msi`) - Windows Installer пакет для корпоративного развёртывания

### Linux
- **AppImage** (`.AppImage`) - portable приложение, не требует установки
- **DEB package** (`.deb`) - пакет для Debian/Ubuntu и совместимых дистрибутивов

### macOS
- **ZIP** (`.zip`) - архив с `.app` bundle для Apple Silicon (M1/M2/M3)

### Android
- **APK** (`.apk`) - Android приложение

### Docker
Образ публикуется в GitHub Container Registry со следующими тегами:
- `latest` - всегда последняя версия
- `X.Y.Z` - конкретная версия (для stable релизов)
- `sha-abcdefg` - тег с хешем коммита (для dev релизов)

## Версия в коде

### Build-time переменные

Версия и метаданные сборки передаются в код через переменные окружения во время компиляции:

| Переменная | Описание | Пример |
|------------|----------|--------|
| `ORIGA_VERSION` | Полная версия приложения | `0.2.0` или `0.1.3-abc1234` |
| `ORIGA_COMMIT` | Полный SHA коммита | `abc1234567890def...` |
| `ORIGA_BUILD_DATE` | Дата сборки в формате YYYY-MM-DD | `2025-01-15` |

### Использование в Rust коде

```rust
pub fn get_version() -> &'static str {
    option_env!("ORIGA_VERSION").unwrap_or("dev")
}

pub fn get_commit() -> &'static str {
    option_env!("ORIGA_COMMIT").unwrap_or("unknown")
}

pub fn get_build_date() -> &'static str {
    option_env!("ORIGA_BUILD_DATE").unwrap_or("unknown")
}
```

### Отображение в UI

Информация о версии доступна пользователю:
- **Путь**: Страница `/profile` → "Настройки приложения" → "О приложении"
- **Отображается**: версия, хеш коммита, дата сборки

## Примеры сценариев

### Сценарий 1: Первый stable релиз

```bash
# Проект только начат, тегов ещё нет
# Создаём первый релиз

git tag v0.1.0
git push origin v0.1.0

# → Релиз 0.1.0
# → Docker образ с тегом 0.1.0
# → Dev-сборки будут начинаться с 0.1.1-*
```

### Сценарий 2: Обычная разработка (dev сборка)

```bash
# Работаем над кодом, тег v0.1.0 существует

# Вносим изменения
echo "// new feature" >> src/main.rs
git add .
git commit -m "feat: добавить новую функцию"
git push origin master

# → Автоматически обновляется релиз "latest"
# → Версия: 0.1.1-abc1234 (где abc1234 - sha коммита)
# → Docker образ с тегом latest
```

### Сценарий 3: Minor версия (новая функциональность)

```bash
# Накоплено достаточно изменений для minor релиза
# Текущий последний тег: v0.1.5

git tag v0.2.0
git push origin v0.2.0

# → Релиз 0.2.0 (stable)
# → Docker образ с тегом 0.2.0 и latest
# → Следующие push в master → 0.2.1-{sha}
```

### Сценарий 4: Patch версия (исправление багов)

```bash
# Обнаружен критический баг после релиза 0.2.0
# Исправляем и выпускаем patch версию

git checkout master
git pull origin master
# исправляем баг
git add .
git commit -m "fix: критический баг"
git push origin master

# Убеждаемся что исправление работает в dev-сборке
# Затем создаём patch релиз

git tag v0.2.1
git push origin v0.2.1

# → Релиз 0.2.1 (stable)
```

### Сценарий 5: Major версия (breaking changes)

```bash
# Готовимся к выпуску версии с breaking changes
# Текущая версия: 0.5.3

git tag v1.0.0
git push origin v1.0.0

# → Релиз 1.0.0 (stable)
# → Новая major версия
```

## Checklist перед релизом

Перед созданием stable релиза рекомендуется проверить:

- [ ] **Все тесты проходят**
  ```bash
  cargo test
  npm test  # e2e тесты
  ```

- [ ] **Clippy без ошибок**
  ```bash
  cargo clippy -- -D warnings
  ```

- [ ] **Форматирование кода**
  ```bash
  cargo fmt -- --check
  ```

- [ ] **Документация актуальна** (опционально)
  - README.md
  - CHANGELOG.md (если ведётся)
  - AGENTS.md

- [ ] **Версии зависимостей** (опционально)
  - Проверить `cargo outdated`
  - Обновить критические зависимости

**Примечание:** Ручное обновление версии в коде не требуется - система автоинкрементного версионирования обрабатывает это автоматически.

## Дополнительная информация

### Docker образы

Docker образы публикуются в GitHub Container Registry:

```bash
# Stable версия
docker pull ghcr.io/username/origa:0.2.0

# Latest (dev)
docker pull ghcr.io/username/origa:latest
```

### Скачать релизы

- **GitHub Releases**: https://github.com/username/origa/releases
- **Stable**: релизы с версией (0.2.0, 0.1.0, ...)
- **Latest Build**: релиз с тегом `latest`

### Troubleshooting

**Проблема:** Релиз не создаётся

**Решение:**
1. Проверь что workflow успешно завершился в Actions
2. Убедись что тег в правильном формате: `v0.2.0` (не `0.2.0`)
3. Проверь права доступа для GitHub Actions

**Проблема:** Неправильная версия в dev-сборке

**Решение:**
1. Проверь что теги синхронизированы: `git fetch --tags`
2. Убедись что последний тег в формате `v*.*.*`

**Проблема:** Артефакты не загрузились

**Решение:**
1. Проверь логи workflow в GitHub Actions
2. Убедись что достаточно места в релизе (лимит 2GB)
3. Проверь права `GITHUB_TOKEN`
